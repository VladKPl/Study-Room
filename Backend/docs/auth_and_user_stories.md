# RBAC и пользовательские сценарии

## Роли

- `guest`: анонимный пользователь, доступ только к чтению публичных курсов.
- `student`: зарегистрированный пользователь, может начинать курс.
- `author`: пользователь с правами автора, может создавать курсы и управлять только своими курсами.
- `admin`: полный доступ к модерации, включая блокировку (`ban`) и полное удаление (`hard delete`).

## Матрица прав

| Действие | guest | student | author | admin |
|---|---|---|---|---|
| Список опубликованных курсов (`GET /api/v1/courses`) | да | да | да | да |
| Просмотр карточки курса (`GET /api/v1/courses/{id}`) | да | да | да | да |
| Создать курс (`POST /api/v1/courses`) | нет | нет | да | да |
| Начать курс (`POST /api/v1/courses/{id}/start`) | нет | да | да | да |
| Мягкое удаление курса (`DELETE /api/v1/courses/{id}`) | нет | нет | да | да |
| Скрыть курс (`PATCH /api/v1/courses/{id}/hide`) | нет | нет | да | да |
| Забанить курс (`PATCH /api/v1/courses/{id}/ban`) | нет | нет | нет | да |
| Полное удаление курса (`DELETE /api/v1/courses/{id}/hard-delete`) | нет | нет | нет | да |

## Пользовательские сценарии

1. Как `guest`, я хочу искать курсы и смотреть карточки, чтобы оценить контент до регистрации.
2. Как `student`, я хочу начинать курс, чтобы приступить к обучению после регистрации.
3. Как `author`, я хочу создавать курсы, чтобы публиковать собственный контент.
4. Как `author`, я хочу скрывать курс из выдачи и делать мягкое удаление, чтобы управлять своим каталогом.
5. Как `admin`, я хочу банить и удалять курсы, чтобы модерировать платформу.

## Технические заметки

- Авторизация строится на заголовке `Authorization: Bearer <access_token>`.
- Если токен не передан, пользователь считается `guest`.
- Для операций `author` проверяется владение курсом (`course.author_id == current_user.id`).
- Роль `student` назначается по умолчанию при регистрации.
- Поддерживается вход через Google OAuth (`/api/v1/auth/google/login`, `/api/v1/auth/google/callback`).

## Запланировано

- Самостоятельное повышение роли `student -> author` через отдельный бизнес-процесс.
